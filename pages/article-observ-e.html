OBSERV-E: Building Assistive Robotics as a Systems Problem, Not a Hardware Problem

Most assistive tech projects focus on the device. OBSERV-E started as three separate robot ideas before I realized the real design challenge was the signal path between them and the person who needs help.

Where it started: three separate robots

When I first started sketching out OBSERV-E, I had three ideas sitting in three separate folders. GRVI was going to be a ground robot that guides visually impaired users through indoor spaces. DRVI was a drone concept for aerial environmental awareness — scouting ahead or covering spaces where ground robots can't reach. HRVI was the wearable receiver that would translate robot signals into something the user could feel.

The problem: I was treating them like three products. Each had its own architecture sketch. Each was going to talk to a phone app. Each had a slightly different philosophy about what the user experience should feel like.

That's when I hit the real design question: if a person with low vision needs guidance, what's the critical path between the robot sensing something and that person receiving useful feedback? It's not the robot. It's not the app. It's the latency and reliability of the signal chain between them.

The architectural shift: cutting out the web app

The original architecture had a familiar shape: robot → web app → phone → user. That's the standard IoT pattern. It's also a chain with multiple points of failure and unpredictable latency. For something like obstacle detection or navigation guidance, "check your phone" is a terrible interaction model.

The shift I made was moving to direct robot-to-user Bluetooth signaling — bypassing the web app entirely for the critical feedback path. The HRVI receiver becomes the endpoint. The robot sends signals that get translated into haptic patterns the user feels immediately, without any server round-trip.

Removes server-side latency from the primary feedback path
Eliminates Wi-Fi dependency for the most critical interaction: guidance cues
Puts the experience in the user's hands — literally — rather than on a screen
The web app becomes a configuration and monitoring layer, not the signal path

This is a pattern I see repeatedly in well-designed assistive systems: the key user interaction should not depend on network reliability. Bluetooth's range and pairing model make it the right layer for real-time feedback.

Why the service dog model matters

The framing shift that clarified everything: I started thinking about OBSERV-E not as a robot that assists visually impaired users, but as a system that behaves like a trained service animal — one that senses the environment, interprets what's relevant, and communicates directly with the person it's guiding.

A guide dog doesn't send you a push notification. It steers you. The feedback is immediate and physical. HRVI's role in the ecosystem is to translate GRVI's perception into that kind of direct, tactile signal. Different haptic patterns for "obstacle ahead," "clear to move," and "stop" — without the user needing to look at anything.

DRVI enters the picture for scenarios where ground-level perception isn't enough: navigating unfamiliar spaces, scouting ahead at intersections, or providing aerial awareness in environments where GRVI's camera has limited field of view. The ecosystem model means these robots can coordinate — they're not competing products, they're a sensing system at different layers.

What I'm still working through

The honest answer is that OBSERV-E is in an early-architecture phase. The system design decisions are clearer than they were when I started, but there's significant implementation work ahead — perception pipelines, Bluetooth protocol design for the HRVI receiver, and the coordination logic between GRVI and DRVI.

What's been most valuable so far: forcing myself to define the ecosystem as one thing instead of three separate projects changed how I think about the architecture, the pitch, and the user experience. The hardest part of building assistive technology isn't the sensing — it's designing the interaction model so the technology disappears and the person can focus on living.

I'm building OBSERV-E out of Illinois Tech Robotics with the goal of getting a working GRVI prototype into structured user testing. If you work on assistive systems, robotics, or accessibility, I'd genuinely value a conversation.
