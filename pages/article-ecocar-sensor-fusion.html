Sensor Fusion at EcoCAR: What Real-Time Validation Actually Looks Like

Writing requirements is one thing. Writing requirements that actually get tested — against real LiDAR, radar, and camera data running in RTMaps on a Chevy Blazer EV — is a completely different discipline.

What the EcoCAR EV Challenge actually is

EcoCAR is the U.S. Department of Energy's premier collegiate automotive engineering competition. The challenge gives a team of students a stock vehicle — in our case, a Chevy Blazer EV — and asks them to redesign and extend it with connected automated vehicle capabilities across a four-year competition cycle.

My role was Sensor Fusion Lead, which meant owning the perception and monitoring layer: how camera, radar, and LiDAR data get combined into something the vehicle's autonomy stack can actually use. It also meant writing the verification and validation requirements that define what "correct behavior" means for that layer — and making sure those requirements were testable.

"Testable" is doing a lot of work in that sentence.

The gap between requirements and real hardware

When you write a requirement like "the lead vehicle detection module shall identify the nearest forward vehicle within 50ms of detection," it sounds precise. The gap shows up when you try to test it.

Real sensor data is messy. LiDAR returns point clouds that need to be interpreted. Radar sees through fog but struggles with static objects. Camera-based detection varies dramatically with lighting. When all three sensors are running simultaneously through RTMaps, each with its own timestamping, synchronization becomes a core engineering problem rather than a footnote.

Sensors report at different frequencies — camera at 30Hz, LiDAR at 10Hz, radar at 20Hz. Fusion requires explicit time-alignment.
A requirement written without understanding timestamp drift will produce test cases that pass in simulation and fail on hardware.
RTMaps makes the timing explicit — you can see the actual latency distribution of your fusion pipeline in the tool's diagram view.
Driver attention monitoring adds a human-factors layer: the system must distinguish intentional behavior from distraction, which is a classification problem with real safety implications.

Working through those gaps — between the clean requirement on paper and the signal behavior on hardware — taught me more about embedded systems engineering than any course could.

RTMaps as an engineering environment

RTMaps (Real-Time Multisensor Applications) is Intempora's middleware for sensor fusion and real-time embedded systems. It's graphical — you build processing pipelines by connecting blocks, which makes the data flow explicit and reviewable. For a student team, this is valuable: you can show a non-expert exactly what data goes where.

The C++ components I implemented plugged into this pipeline as processing blocks. Lead vehicle detection required taking fused camera and radar data, applying detection logic, and producing a vehicle state estimate that the vehicle's control layer could consume — all with defined latency bounds.

The most useful thing RTMaps does: it lets you replay recorded sensor data through your pipeline. That means you can test a new algorithm version against the exact same real-world data, without needing the vehicle present. This is how automotive-grade V&V teams work — and learning it in a student competition context gave me hands-on exposure to the pattern.

What I hadn't expected: the debugging workflow for real-time systems is fundamentally different from debugging application code. Race conditions, timing-dependent failures, and sensor-specific edge cases don't reproduce reliably. You learn to instrument first and log everything.

What sensor fusion taught me about engineering judgment

The most counterintuitive lesson: more sensors does not mean better perception. It means more data to reconcile — and more ways to be confidently wrong.

When camera and radar disagree about whether there's a vehicle ahead, you need a decision rule. Naive fusion (just average them) produces worse results than either sensor alone in certain conditions. The engineering judgment is knowing when to trust which sensor, how to weight disagreements, and how to flag uncertainty rather than suppress it.

This is the same pattern that showed up in my RL lane-keeping simulation work: a more complex sensor configuration doesn't automatically produce better policy learning. The sensor stack needs to be matched to the task and the architecture, not just maximized.

EcoCAR's structured competition environment — with OEM-grade requirements expectations, peer review of test plans, and real hardware sessions — gave me a level of engineering rigor that I try to bring into every project. Requirements should have traceability. Tests should be designed before implementation. Failure modes should be documented, not just fixed.
