The demo gap

There's a specific failure mode in AI projects that I started calling "demo mode": the system works in the demo. It produces impressive outputs in the exact context it was designed to show. But when someone slightly outside that context uses it — different input, different environment, different network — it breaks in ways the demo never revealed.

I spent the summer of 2025 as an AI Software Development Intern at OfficePro, Inc., where I ran into this pattern repeatedly from both sides: building new AI prototypes and maintaining production systems where the demo gap had already been found, usually by a real user.

The things I learned weren't about model architecture or prompt engineering. They were about system design, operational reality, and the difference between "it works on my machine" and "it works for the person paying for it."

Building Azure-backed AI training tools

One of my primary projects was developing AI instructor and training-support prototypes using Azure Cognitive Services and Azure Speech Services. The concept: an interactive AI that could support a 50+ computer training fleet at OfficePro's client sites — walking users through procedures, answering questions, and adapting to different skill levels.

What this actually required, in practice:

Azure Functions as the orchestration layer — keeping the AI logic serverless and scalable, rather than running it on a machine that might be rebooted or disconnected mid-session.
Speech Services latency budgeting — TTS and STT add real latency. For an interactive training context, >1.5s response time breaks the conversational flow. Optimizing the pipeline to stay under that threshold was unglamorous but critical work.
Python + PowerShell for fleet automation — the 50+ computer fleet needed consistent configuration. Automation scripting that works reliably across a heterogeneous Windows fleet is not the same as scripting that works on your dev machine.
Error states are user experiences — when the speech service times out, the user sees something. Designing graceful degradation — falling back to text, surfacing a clear error message, not just silently failing — is part of the product, not an afterthought.

The prototype phase is where you figure out if the idea works at all. The hardening phase is where you figure out if it works for humans in the real world.

Production lessons from AIHeadShotMasters

The second major thread of the internship was contributing to production workflows on AIHeadShotMasters — an AI headshot generation platform. This was not prototype work. Real users, real payments, real consequences when things broke.

The stack: TypeScript on the frontend, Supabase for database and auth, Vercel for CI/CD, Stripe for payments. When issues appeared in production — and they did — tracing them required understanding all four layers simultaneously.

Some specific things that production taught me that prototypes never would:

Database migrations in production are different — running a schema change against a Supabase instance with live data requires a different level of care than running it locally. Row locking, migration atomicity, and rollback paths matter.
Stripe webhook idempotency — payment webhooks can be delivered more than once. The handler needs to be idempotent — processing the same event twice should not charge a customer twice or duplicate a record. This is a class of bug that a demo will never surface.
Vercel preview deployments are not production — environment variables, edge function behavior, and cold start timing can differ between preview and production. A fix that works in preview can fail in production in subtle ways.
Debugging someone else's production system requires reading the code before writing any — the most effective thing I did was spend time understanding what the system was doing before I changed anything.

The mindset shift: systems over models

The biggest takeaway from the summer wasn't technical — it was a change in how I think about AI projects. The model is one component. The system around it — the orchestration, the infrastructure, the error handling, the operational monitoring, the user-facing behavior when things go wrong — is the product.

This sounds obvious in retrospect, but it changes how you approach a project from the start. A prototype that's designed to become a production system looks different from one that's designed to generate a demo. You build in observability earlier. You think about failure modes before they occur. You design the degraded-state experience alongside the happy path.

AILA — the AI Leadership Avatar I built for the Illinois Tech Leadership Academy — benefited directly from these lessons. The voice system, the Ollama integration, the fallback behavior when TTS is unavailable — all of those were designed with the production mindset, not just "does it work in the demo."
